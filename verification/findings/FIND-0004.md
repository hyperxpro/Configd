# FIND-0004: Signature verification uses different encoding paths — likely broken

## Severity
Critical

## Discovered by
blue-code-audit-config-store

## Confirmed by
Main auditor (pending: need to trace ConfigStateMachine.signCommand + DeltaApplier.buildVerificationPayload)

## Location
ConfigStateMachine.java:205, DeltaApplier.java:164-175

## Description
The leader signs raw Raft command bytes in `ConfigStateMachine.signCommand()`. The edge verifier in `DeltaApplier.buildVerificationPayload()` reconstructs the payload by re-encoding from deserialized mutations via `CommandCodec`. These code paths may produce different byte sequences:

1. Batch deltas aggregate multiple log entries — the signed payload is per-entry, but verification is per-delta
2. The `ConfigMutation.Put` record defensively clones value bytes, and re-encoding from cloned data goes through a different serialization path
3. Any difference in byte ordering, padding, or encoding between the two paths means signature verification systematically fails or is never actually verified

## Reproduction
Requires tracing both code paths with a batch mutation to confirm byte divergence.

## Impact
If signatures systematically fail, either:
- The verification is silently skipped (security bypass)
- All deltas are rejected (system broken)
- Only batch deltas fail (intermittent, hard to detect)

## Root Cause
Sign-then-verify should operate on the same byte representation. The current design signs the input representation and verifies against a reconstructed representation.

## Fix
Sign the delta's serialized form directly, not the individual Raft commands. Or: include the signed bytes alongside the delta so the verifier can check the original bytes.

## Re-verified by
Build passes. Root cause confirmed: single-mutation batch commands were signed as batch format [0x03][1][...] 
but verified as single PUT format [0x01][...], causing systematic signature mismatch.

Fix: both sign (ConfigStateMachine.canonicalize()) and verify (DeltaApplier.buildVerificationPayload()) 
now normalize to batch-canonical form. Three regression tests added:
1. `signatureVerifiesWithPublicKey` — verifies canonical form, asserts raw form does NOT verify
2. `signatureVerifiesForBatchCommand` — batch roundtrip
3. `singleMutationBatchSignatureMatchesSinglePutCanonical` — the exact FIND-0004 scenario
4. `find0004_singleMutationSignedByLeaderVerifiesAtEdge` — end-to-end leader→edge

## Doc/Spec Updates
Security model should document what is signed and at what granularity.

## Status
Fixed — canonical batch normalization on both sign and verify paths
