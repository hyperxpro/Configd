# FIND-0006: FileStorage.put() is not crash-safe (no atomic rename)

## Severity
Critical

## Discovered by
blue-code-audit-remaining (FIND-C09)

## Confirmed by
Main auditor (FileStorage.java:46-59 uses truncate-then-write)

## Location
FileStorage.java:46-59

## Description
`FileStorage.put()` uses truncate-then-write without atomic rename. The sequence is:
1. Open file for writing (truncates existing content)
2. Write new content
3. fsync

A crash between step 1 (truncation) and step 3 (fsync) leaves a corrupt or empty file. For Raft persistent state (`currentTerm`, `votedFor`), this violates Raft safety invariants — a node could restart with term=0 and votedFor=null even after voting in a higher term.

## Impact
On crash recovery, a node could:
- Vote for two different candidates in the same term (split vote → two leaders)
- Regress its term to 0 (accept stale messages from any term)

## Fix
Use write-to-temp + fsync + atomic rename pattern:
```java
Path tmp = path.resolveSibling(path.getFileName() + ".tmp");
Files.write(tmp, data);
// fsync the file
try (FileChannel ch = FileChannel.open(tmp, StandardOpenOption.WRITE)) {
    ch.force(true);
}
// Atomic rename
Files.move(tmp, path, StandardCopyOption.ATOMIC_MOVE);
// fsync the directory
try (FileChannel dir = FileChannel.open(path.getParent())) {
    dir.force(true);
}
```

## Status
Fixed — FileStorage.put() now uses write-to-tmp + fsync + atomic rename + directory sync. Regression test `putUsesAtomicRename` added.
