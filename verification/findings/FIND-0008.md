# FIND-0008: FanOutBuffer ring buffer has TOCTOU race and gap bug

## Severity
Critical

## Discovered by
blue-code-audit-distribution (FanOutBuffer.java:33-41, 49)

## Confirmed by
Main auditor (pending independent verification)

## Location
FanOutBuffer.java:33-41, FanOutBuffer.java:49

## Description
Two issues in FanOutBuffer:

1. **TOCTOU race**: The "lock-free" ring buffer's append operation writes to `ring.set()` and then updates `head` with a volatile write. These are two separate operations. A concurrent reader could see the updated `head` but read the old value at the new position (or the other way around if the JIT reorders the non-volatile array write after the volatile head write).

2. **Gap bug**: The `deltasSince()` filter logic uses comparison that skips deltas spanning the requested version, causing permanent gaps for consumers.

## Impact
1. Consumers can read garbage/stale data from the ring buffer
2. Consumers can miss deltas permanently, causing silent divergence from the leader's state

## Fix
1. Use AtomicReferenceArray for the ring, or use a proper publication mechanism
2. Fix the version comparison in deltasSince()

## Re-assessment
The current implementation ALREADY uses `AtomicReferenceArray` (volatile semantics per slot) and 
volatile `head`/`tail` fields. The happens-before ordering is: `ring.set(slot, delta)` → 
`head = head + 1` (volatile write), and reader: `head` (volatile read) → `ring.get(slot)` 
(volatile read). By transitivity, the reader is guaranteed to see the published delta. 
The "gap bug" in `deltasSince()` (`fromVersion >= fromVersion`) is correct for the versioning 
model — the `DeltaApplier` provides exact version matching as defense-in-depth.

Existing concurrent stress tests (`ConcurrentReadDuringWrite`, 10K writes / 4 readers) confirm 
no data corruption under concurrent access.

## Status
Already mitigated — AtomicReferenceArray provides safe publication; DeltaApplier provides gap detection defense-in-depth
