# FIND-0001: triggerSnapshot() captures uncommitted config state

## Severity
Critical

## Discovered by
blue-code-audit-consensus

## Confirmed by
Main auditor (independent code review of RaftNode.java:301-316)

## Location
RaftNode.java:312

## Description
`triggerSnapshot()` serializes the current in-memory `clusterConfig` as the snapshot's config data. However, the snapshot is taken at `lastApplied` index, while `clusterConfig` reflects the latest config entry in the *entire* log — including uncommitted entries (per Raft §4.1, nodes use the latest config regardless of commit status).

If there is an uncommitted config change beyond `lastApplied`, the snapshot records a config that does not correspond to the state at `lastApplied`.

## Reproduction
1. 3-node cluster with leader L, followers F1, F2
2. L commits entries through index 100
3. L proposes a joint config change at index 101 (not yet committed — requires quorum)
4. L calls `triggerSnapshot()` before index 101 commits
5. Snapshot is at index 100 but contains the joint config from index 101
6. F2 is far behind and needs InstallSnapshot
7. F2 installs the snapshot: it now has state at index 100 but believes it's in joint consensus
8. F2 uses joint quorum rules for elections, but the joint config was never committed

## Impact
A follower could adopt a cluster configuration that was never committed. This could:
- Require dual-majority quorums from a config that was abandoned
- Prevent elections from succeeding if the new voter set includes unreachable nodes
- In the worst case, cause the cluster to become stuck if the uncommitted config added nodes that don't exist

## Root Cause
The snapshot config should be derived from the log at `lastApplied`, not from the current in-memory `clusterConfig`. The `recomputeConfigFromLog()` method scans the full log, but there is no equivalent that scans only up to a given index.

## Fix
```java
// In triggerSnapshot(), scan log up to appliedIndex for config, not use current clusterConfig
private ClusterConfig configAtIndex(long index) {
    for (long i = Math.min(index, log.lastIndex()); i > log.snapshotIndex(); i--) {
        LogEntry entry = log.entryAt(i);
        if (entry != null && isConfigChangeEntry(entry.command())) {
            return deserializeConfigChange(entry.command());
        }
    }
    // Fall back to snapshot config or initial config
    if (latestSnapshot != null && latestSnapshot.clusterConfigData() != null
            && isConfigChangeEntry(latestSnapshot.clusterConfigData())) {
        return deserializeConfigChange(latestSnapshot.clusterConfigData());
    }
    var allVoters = new HashSet<>(config.peers());
    allVoters.add(config.nodeId());
    return ClusterConfig.simple(allVoters);
}
```

## Re-verified by
Build passes (20,161+ tests). Fix implemented in RaftNode.java via new `configAtIndex(long index)` method.

## Doc/Spec Updates
TLA+ spec should be updated to include a TakeSnapshot action that captures config at the applied index, not the latest config.

## Status
Fixed — RaftNode.java `triggerSnapshot()` now uses `configAtIndex(appliedIndex)` instead of `clusterConfig`
