# FIND-0002: ReadIndex not linearizable under leadership change race

## Severity
Critical

## Discovered by
blue-code-audit-consensus

## Confirmed by
Main auditor (independent analysis of ReadIndexState + RaftNode interaction)

## Location
RaftNode.java:330-341, ReadIndexState.java

## Description
The ReadIndex protocol has a linearizability gap. After `readIndex()` records the commit index and `confirmPendingReads()` confirms leadership via heartbeat, the `isReadReady()` check only verifies:
1. Leadership was confirmed (past tense)
2. `lastApplied >= readIndex`

It does NOT verify that this node is still the leader at the time `isReadReady()` returns true. If the leader is deposed between confirmation and the caller serving the read, the read may be stale.

## Reproduction
1. Leader L calls `readIndex()`, gets readId with commitIndex=100
2. `tickHeartbeat()` fires, heartbeats confirm leadership, `confirmPendingReads()` marks reads as confirmed
3. Network partition isolates L from the quorum
4. New leader L' is elected and commits entries 101-105
5. L hasn't yet detected the partition (CheckQuorum hasn't fired)
6. Caller on L calls `isReadReady(readId)` — returns true
7. Caller serves the read from L's state machine at index 100
8. This read misses entries 101-105 that were committed before it was served

The window is: heartbeat interval (during which CheckQuorum hasn't fired) minus time since last heartbeat.

## Impact
Linearizable read guarantee is violated. A client can observe stale data from a deposed leader. This directly contradicts consistency-contract.md §1 (INV-L1).

## Root Cause
`isReadReady()` is a pure function on recorded state — it doesn't re-verify leadership. The protocol assumes that between confirmation and read serving, no leadership change can occur. This is only true if the caller serves the read within a single tick.

## Fix
Add a role check to `isReadReady()`:
```java
public boolean isReadReady(long readId) {
    if (role != RaftRole.LEADER) {
        return false; // No longer leader — read is invalid
    }
    return readIndexState.isReady(readId, log.lastApplied());
}
```
This is necessary but not sufficient — even with this check, there's a TOCTOU between checking `role` and serving the read. The correct approach is to have `isReadReady()` return a `ReadResult` that includes the role assertion, and the caller must serve the read immediately within the same tick.

## Re-verified by
Build passes. Fix implemented: `isReadReady()` now checks `role != RaftRole.LEADER` before returning true.

## Doc/Spec Updates
ReadIndex protocol should be documented in consistency-contract.md with the exact linearizability window guarantee.

## Status
Fixed — RaftNode.java `isReadReady()` now includes leader role check
