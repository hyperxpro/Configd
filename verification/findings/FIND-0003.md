# FIND-0003: ReadResult allocates on every cache hit — "zero allocation" claim is false

## Severity
Critical

## Discovered by
blue-code-audit-config-store

## Confirmed by
Main auditor (code trace: VersionedConfigStore.get() -> ReadResult.foundReusable() -> new ReadResult())

## Location
ReadResult.java:56-58, VersionedConfigStore.java:195, LocalConfigStore.java (caller)

## Description
Multiple docs claim "zero allocation on read path":
- HamtMap.java:17 — "Read path: zero allocation"
- VersionedConfigStore.java:19 — "zero allocation on the read path"
- performance.md §1 — "0 B/op" for cache miss, "~48 B/op" for cache hit
- architecture.md §3 — "Zero allocation. Zero locks."

The HamtMap claim is technically correct — `HamtMap.get()` allocates nothing.
But the end-to-end read path through `VersionedConfigStore.get()` calls `ReadResult.foundReusable()` which allocates a new `ReadResult` object (~24-48 bytes) on every cache hit.

The method name `foundReusable` is itself misleading — it was inherited from a removed ThreadLocal flyweight pattern and does not reuse anything.

## Reproduction
Run `VersionedStoreReadBenchmark.getHit` with `-prof gc`:
```
VersionedStoreReadBenchmark.getHit:gc.alloc.rate.norm  10000  avgt  ~48 B/op
```
The 48 B/op allocation is documented in performance.md but contradicted by the "zero allocation" claims in multiple other documents.

## Impact
- Documented contract violation: external consumers relying on the "zero allocation" guarantee will measure ~48 B/op
- At 1M reads/sec: ~48 MB/sec of short-lived allocations, all in ZGC young generation
- The allocation is operationally acceptable (ZGC handles this trivially), but the documentation is wrong

## Root Cause
The ThreadLocal flyweight was correctly removed (aliasing hazard), but the documentation across multiple files was not updated to reflect the new allocation behavior.

## Fix
1. Update all documentation to say "zero allocation on miss, ~48 B/op on hit"
2. Rename `foundReusable` to `found` (it already has an alias)
3. Consider: pre-allocating a thread-local `ReadResult` with copy-on-read semantics, or using a record with escape analysis hints

## Re-verified by
Build passes. `foundReusable()` deprecated in favor of `found()`. All callers updated. Javadoc and architecture.md corrected.

## Doc/Spec Updates
- performance.md §1 already documents the ~48 B/op — no change needed
- architecture.md §3 — corrected to "Zero allocation on miss, ~24 B on hit"
- VersionedConfigStore.java Javadoc — corrected to document per-hit allocation
- HamtMap.java Javadoc is correct (HAMT itself is zero-alloc) — no change

## Status
Fixed — `foundReusable()` deprecated, callers migrated to `found()`, doc claims corrected
