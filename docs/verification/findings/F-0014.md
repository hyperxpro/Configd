# F-0014: FanOutBuffer.deltasSince() can silently miss deltas under concurrent access

## Severity
P2

## Phase
V3

## Component
configd-distribution-service:FanOutBuffer.java:43

## Invariant or target violated
INV-S1 (Staleness bounds) — edge nodes may experience unbounded staleness if deltas are silently dropped

## Reproducer
1. Configure a FanOutBuffer with a small ring buffer (e.g., 16 slots) and a single writer producing deltas at high frequency.
2. Have a slow consumer call `deltasSince()` with a version that is near the tail of the buffer.
3. The consumer reads `tail` first (volatile read), then before it reads `head`, the writer overwrites multiple slots — advancing `head` and `tail` past the consumer's position.
4. When the consumer reads `head` (second volatile read), it sees a newer value. It then iterates slots that now contain newer deltas instead of the expected ones.
5. The `fromVersion` filter inside the loop silently drops these mismatched deltas because their `fromVersion()` does not match the expected version.
6. The consumer receives an incomplete delta set with no error indication, causing it to miss updates.

## Evidence
`deltasSince()` reads `tail` before `head`. These are two separate volatile reads with no compound atomicity guarantee. If the writer overwrites slots between those two reads (head incremented, tail advanced, old slot overwritten), the reader iterates slots containing new deltas instead of the expected ones. The `fromVersion` filter silently drops these mismatched deltas, causing the consumer to miss updates without any error indication. The consumer's version cursor does not advance, but it also does not know that deltas were missed — it simply receives fewer deltas than expected.

## Root cause
`deltasSince()` reads `tail` before `head`. If the writer overwrites slots between those two volatile reads (head incremented, tail advanced, old slot overwritten), the reader iterates slots containing new deltas instead of the expected ones. The `fromVersion` filter silently drops these mismatched deltas, causing the consumer to miss updates without error indication.

## Proposed fix
Documented as known limitation. The `DeltaApplier`'s version continuity check (`delta.fromVersion()` must equal `currentVersion`) provides defense-in-depth — a gap triggers full snapshot resync. No code change needed but the behavior should be documented.
