# F-0013: Snapshot key length overflow for keys > 65535 bytes

## Severity
P2

## Phase
V3

## Component
configd-config-store:ConfigStateMachine.java:270

## Invariant or target violated
Snapshot integrity — round-trip through snapshot()/restoreSnapshot() must preserve all data

## Reproducer
1. Insert a key into the config store that is longer than 65535 bytes (e.g., a 70000-byte key string).
2. Trigger a snapshot via the normal Raft snapshotting mechanism.
3. Observe that `snapshot()` calls `putShort((short) keyBytes.length)` which silently truncates the length to `keyBytes.length & 0xFFFF`.
4. Restart the node and trigger snapshot restore.
5. `restoreSnapshot()` reads the truncated length via `Short.toUnsignedInt()`, reads the wrong number of bytes, and either produces a corrupted key or throws an exception due to buffer underflow/misalignment.

## Evidence
In `ConfigStateMachine.snapshot()` at line 270, key lengths are encoded via `putShort((short) keyBytes.length)`. Java's cast to `short` silently truncates values above 32767 (signed) or 65535 (unsigned). The restore side uses `Short.toUnsignedInt()` which correctly handles values up to 65535, but any key with a byte length exceeding 65535 would have its length silently truncated during snapshot, corrupting the snapshot format and all subsequent entries.

## Root cause
`putShort((short) keyBytes.length)` silently truncates keys > 65535 bytes. The restore side used `Short.toUnsignedInt()` which handles up to 65535, but keys > 65535 would be corrupted.

## Proposed fix
Changed to `putInt`/`getInt` for key length encoding. Updated both `snapshot()` and `restoreSnapshot()`.
