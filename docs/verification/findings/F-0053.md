# F-0053: ConfigStateMachine.restoreSnapshot Lacks Bounds Checks (OOM / Malformed-Snapshot)

**Severity:** P2 (Medium)
**Phase:** V8 — Security Audit
**Status:** Closed — fix: ConfigStateMachine.restoreSnapshot bound-checks entryCount, keyLen, valueLen against MAX_SNAPSHOT_* and rejects truncated payloads with typed IllegalArgumentException; test: io.configd.store.ConfigStateMachineTest$SnapshotBoundsCheck::{restoreRejectsAbsurdEntryCount, restoreRejectsNegativeEntryCount, restoreRejectsNegativeKeyLength, restoreRejectsAbsurdKeyLength, restoreRejectsTruncatedKeyBytes, restoreRejectsAbsurdValueLength}

## Summary

`ConfigStateMachine.restoreSnapshot(byte[])` reads `entryCount`, `keyLen`,
and `valueLen` as 32-bit signed ints directly from the buffer and
allocates based on the raw values without any bounds check. A peer that
ships a malicious or corrupted InstallSnapshot payload can trigger either
an `OutOfMemoryError` (large positive length) or `NegativeArraySizeException`
(negative length), in both cases taking the receiving node down during
a critical recovery code path.

Unlike `CommandCodec.decodePut` — which enforces `MAX_VALUE_SIZE = 1_048_576`
and `MAX_BATCH_COUNT = 10_000` — the snapshot decoder has no sanity limits.
This is also inconsistent with `TcpRaftTransport` which caps frame length
at 16 MiB.

## Location

- `configd-config-store/src/main/java/io/configd/store/ConfigStateMachine.java:290-320`

```java
ByteBuffer buf = ByteBuffer.wrap(snapshot);
long restoredSequence = buf.getLong();
int entryCount = buf.getInt();             // no bound
for (int i = 0; i < entryCount; i++) {
    int keyLen = buf.getInt();             // no bound
    byte[] keyBytes = new byte[keyLen];    // OOM or NASE
    ...
    int valueLen = buf.getInt();           // no bound
    byte[] value = new byte[valueLen];     // OOM or NASE
    ...
}
```

## Impact

- DoS against a node that is installing a snapshot from a (possibly
  compromised, possibly just faulty) leader.
- Crashing during InstallSnapshot is a correctness hazard: the node has
  typically already truncated its log and now cannot resume.

## Suggested Fix

Apply the same envelope-level limits as `CommandCodec`:

```java
if (entryCount < 0 || entryCount > MAX_ENTRIES) throw ...;
if (keyLen < 0 || keyLen > MAX_KEY_LEN)          throw ...;   // 1024
if (valueLen < 0 || valueLen > MAX_VALUE_SIZE)   throw ...;   // 1_048_576
if (buf.remaining() < keyLen)                    throw ...;   // truncation
if (buf.remaining() < valueLen)                  throw ...;
```

Reject the snapshot cleanly rather than letting the allocator crash the
JVM.

## Regression Test

Property test: feed `restoreSnapshot` with random byte arrays and with
targeted adversarial payloads (huge counts, negative lengths, lengths
exceeding `snapshot.length`). All must throw a typed domain exception,
never `OutOfMemoryError`.
