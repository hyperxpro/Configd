# F-0042: VersionedConfigStore.get() and LocalConfigStore.get() allocate ReadResult on every hit

## Severity
P2

## Phase
V-perf

## Component
- configd-config-store/src/main/java/io/configd/store/VersionedConfigStore.java:196,214
- configd-edge-cache/src/main/java/io/configd/edge/LocalConfigStore.java:106,133
- configd-config-store/src/main/java/io/configd/store/ReadResult.java:44

## Invariant or target violated
docs/performance.md — "Zero allocation on read path steady state"; "VersionedStore getHit end-to-end p99 < 1 ms".

Violates the zero-allocation claim, although the ReadResult Javadoc acknowledges "~24 B" per hit. This contradicts architecture.md §0.1's unconditional zero-alloc target.

## Reproducer
Read the source:
- `VersionedConfigStore.get(...)` returns `ReadResult.found(vv.valueUnsafe(), vv.version())`.
- `ReadResult.found(byte[], long)` body: `return new ReadResult(value, version, true);`.
- `LocalConfigStore.get(...)` has the same pattern.

A JMH run of `VersionedStoreReadBenchmark.getHit -prof gc` will confirm ~24–32 B/op steady-state allocation.

## Evidence
The hit path of both stores unconditionally wraps the `(value, version, found)` tuple in a fresh `ReadResult`. `NOT_FOUND` is correctly cached (documented as zero-alloc on miss), but every hit allocates.

Note the vestigial `ReadResult.foundReusable(...)` (deprecated for removal) suggests an earlier effort at a thread-local reusable result object that was abandoned — the replacement just allocates.

## Root cause
ReadResult is a non-value class carrying three fields and must outlive the caller in some APIs (e.g., gRPC responses), so scalar replacement is unreliable. No pooled / thread-local return slot is used.

## Proposed fix
Three options in increasing order of ambition:
1. **Document and accept** — update docs to say "zero alloc on miss, ~24 B on hit" (matches the existing ReadResult Javadoc). Cheapest. Probably correct choice given the target is "p99 < 1 ms" which is easily met with 24-byte TLAB allocs.
2. **Return `long version` + out-param `byte[][]`** via a primitive-friendly API for hot callers (e.g., `long getVersion(String key, byte[][] out)`). Keep `ReadResult` API for ergonomic callers.
3. Convert `ReadResult` to a JDK 25 **value class** so the JIT can inline and scalar-replace on the stack.

## Status
Closed — fix: adopted the hybrid option from the proposed list (VDR-0001 at `docs/verification/decisions/vdr-0001.md`). `ReadResult` is retained as the ergonomic API and the ~24 B/hit cost is formally accepted and documented in `docs/performance.md` (p99 < 1 ms swamps a TLAB bump allocation). For the throughput-critical internal callers an allocation-free primitive-friendly API was added: `VersionedConfigStore.getInto(String key, byte[] dst, long[] versionOut)` and a mirror on `LocalConfigStore`, returning value length on hit, `-1` on miss, and `-(n+1)` if `dst` is too small. test: `configd-config-store/src/test/java/io/configd/store/VersionedConfigStoreAllocationTest.java` (new, 3 tests): zero-alloc-on-hit, zero-alloc-on-miss, negative-length resize contract.
