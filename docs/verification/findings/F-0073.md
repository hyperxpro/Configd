# F-0073: Consistency-contract invariants INV-M1 and INV-S1 have no runtime assertion via InvariantMonitor

## Severity
P2 (Medium)

## Phase
V8 (4-leg coverage audit)

## Component
- docs/consistency-contract.md §8 (Runtime Assertions table)
- configd-edge-cache/src/main/java/io/configd/edge/StalenessTracker.java
- configd-edge-cache/src/main/java/io/configd/edge/LocalConfigStore.java (and VersionedConfigStore.java)
- configd-observability/src/main/java/io/configd/observability/InvariantMonitor.java

## Invariant or target violated
consistency-contract.md §8 "Runtime Assertions" table asserts that each invariant has a runtime assertion:

| Invariant | Assertion Name | Location |
|---|---|---|
| INV-M1 | `assert_monotonic_read` | Edge read path |
| INV-S1 | `assert_staleness_bound` | StalenessTracker |

The actual `InvariantMonitor.check(...)` call sites (grep across the repo) register these names:
- `sequence_monotonic`, `sequence_gap_free`, `per_key_order` (ConfigStateMachine)
- `single_server_invariant`, `no_op_before_reconfig`, `reconfig_safety`, `log_matching`, `election_safety`, `leader_completeness`, `version_monotonicity`, `state_machine_safety` (RaftNode)

There is **no** `invariantChecker.check("monotonic_read", ...)` or `check("staleness_bound", ...)` anywhere. StalenessTracker does a structural threshold transition (CURRENT→STALE→DEGRADED→DISCONNECTED) but does not route the violation through `InvariantMonitor`, so production metric `invariant.violation.staleness_bound` will never fire. Similarly, the edge read path (`LocalConfigStore`, `VersionedConfigStore`) enforces cursor monotonicity by blocking/returning stale, but does not record an invariant violation when the cursor cannot be satisfied within `monotonic_read_timeout`.

## Evidence
- `StalenessTracker.java:23`: `private static final long STALE_THRESHOLD_MS = 500;` — matches consistency-contract.md §2, but the state transition logic (line 95) does not call `InvariantMonitor.check`.
- `InvariantMonitor.java` is the designated bridge from TLA+ to runtime (Javadoc: "Every TLA+ invariant in the formal model should have a corresponding Java assertion registered here"). The two data-plane invariants (INV-M1, INV-S1) fall outside Raft's InvariantMonitor coverage.

## Impact
In production mode, a monotonic-read violation or a staleness-bound violation will not increment `configd.invariant.violation.*` counters. Alerting configured on those counters (as recommended by consistency-contract.md §8 and ADR-0007 §3 Runtime Assertions) will silently miss data-plane correctness drift.

## Proposed fix
1. Inject `InvariantMonitor` into `StalenessTracker`; on transition from CURRENT to STALE, call `monitor.check("staleness_bound", false, "edge staleness exceeded 500ms: " + ms)`.
2. Inject `InvariantMonitor` into `VersionedConfigStore`/`LocalConfigStore`; when a cursor-bound read cannot be satisfied within `monotonic_read_timeout`, call `monitor.check("monotonic_read", false, "cursor=" + cursor + " local=" + localVersion)`.
3. Add corresponding tests to `InvariantMonitorTest` or the respective component tests.

## Status
Closed — fix: added `InvariantMonitor.assertMonotonicRead(key, seenVersion, newVersion)` (INV-M1) and `assertStalenessBound(local, remote, staleMs, thresholdMs)` (INV-S1) with public name constants; wired into `LocalConfigStore.get(key, cursor)` (fires when `snapshot.version() < cursor.version()`) and `StalenessTracker.isStale(threshold)` (fires when elapsed > threshold); test: `InvariantMonitorTest.MonotonicRead`/`StalenessBound` nests (4 tests each), `LocalConfigStoreTest.MonotonicReadInvariant` and `StalenessTrackerTest.InvariantMonitorWiring` verify both metric counter (`configd.invariant.violation.monotonic_read` / `staleness_bound`) increments and test-mode `AssertionError`.
