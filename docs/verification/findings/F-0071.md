# F-0071: ADR-0009 (Accepted) claims Netty + gRPC + Spring Boot, but code uses plain TCP/SSLSocket/com.sun.net.httpserver

## Severity
P1 (High)

## Phase
V8 (re-audit of 2026-04-14 reconciliation)

## Component
docs/decisions/adr-0009-java21-zgc-runtime.md (Status: Accepted)

## Invariant or target violated
Doc-vs-code reconciliation. ADR-0010 was properly marked Superseded with a note acknowledging that the implementation uses TCP + SSLSocket + `com.sun.net.httpserver`. ADR-0009, however, is still marked **Accepted** yet explicitly references:
- "Virtual threads for I/O-bound operations (**gRPC stream handling**, Raft RPC, client connections)"
- Thread model diagram: "**gRPC Server Threads** ← Netty event loop (platform threads) + virtual thread per RPC for control plane"
- "Why not GraalVM Native Image? ... **Spring Boot (used for control plane API)** has GraalVM support"

None of these technologies exist in the codebase:
- `pom.xml` has zero Netty, gRPC, or Spring dependencies (only Agrona, JCTools, JUnit, JMH, HdrHistogram, Micrometer, jqwik).
- `configd-server/src/main/java/io/configd/server/HttpApiServer.java:3-7` imports `com.sun.net.httpserver.HttpServer/HttpsServer`.
- `configd-transport/src/main/java/io/configd/transport/TcpRaftTransport.java` uses plain `java.net.Socket`/`SSLSocket`.

## Reproducer
```
$ grep -r "grpc\|netty\|spring" /home/ubuntu/Programming/Configd/pom.xml
(no matches)

$ grep -n "com.sun.net.httpserver" /home/ubuntu/Programming/Configd/configd-server/src/main/java/io/configd/server/HttpApiServer.java
3: import com.sun.net.httpserver.HttpExchange;
4: import com.sun.net.httpserver.HttpHandler;
5: import com.sun.net.httpserver.HttpServer;
```

## Evidence
ADR-0010's V8 reconciliation note explicitly names the actual transport (`TcpRaftTransport.java`, `TlsManager.java`, `FrameCodec.java`, `com.sun.net.httpserver.HttpServer`). ADR-0009 was not updated at the same time and still propagates the Netty/gRPC/Spring narrative. Because ADR-0009 is Accepted (not Superseded), a reader would reasonably believe the claims hold.

## Root cause
Phase V8 reconciled ADR-0010 and ADR-0016 but did not cascade the transport/framework corrections to ADR-0009 (runtime) or ADR-0014 (GC strategy — see F-0072).

## Proposed fix
Add a "Note (2026-04-16, V8 re-audit)" block to ADR-0009 analogous to the ADR-0010 note, or update the Thread Model and "Why not GraalVM" subsections in-place to cite the actual stack:
- "gRPC stream handling" → "HTTP/2 stream handling via com.sun.net.httpserver + blocking IO via virtual threads for TCP transport"
- "Netty event loop" → "Virtual-thread-per-connection model (java.net.Socket + SSLSocket)"
- "Spring Boot (used for control plane API)" → "JDK built-in HttpServer (control plane API)"

## Reporter / Reproducer
- Reporter: code-quality-auditor (Phase V8 re-audit, 2026-04-16)
- Independent reproducer: pom.xml + HttpApiServer.java inspection

## Status
Closed — 2026-04-16. ADR-0009 Status changed to "Superseded (see ADR-0022 for the Java 25 upgrade)" with a prepended V8 re-audit note clarifying that the Netty/gRPC/Spring Boot / Agrona / JCTools references in the body are aspirational and not reflected in the codebase.
