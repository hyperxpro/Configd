# F-0011: WAL recovery crashes on truncated trailing entry

## Severity
P1

## Phase
V3

## Component
configd-common:FileStorage.java:139

## Invariant or target violated
Crash recovery must not prevent server restart (availability)

## Reproducer
1. Start a Raft node and issue writes to generate WAL entries.
2. Kill the process (e.g., `kill -9`) during an `appendToLog()` call — specifically after the length header has been written but before the data and CRC are fully flushed.
3. Restart the node.
4. Observe that `readLog()` throws an `IOException` because the remaining bytes in the file are less than `length + 4` (data + CRC) for the last entry.
5. The server fails to start entirely, even though all entries before the truncation point are valid.

## Evidence
In `FileStorage.readLog()` at line 139, when the remaining bytes in the WAL file are fewer than the declared entry length plus the 4-byte CRC, the code throws an `IOException`. This is the wrong behavior for a trailing partial entry — WAL implementations universally expect that the last entry may be incomplete after a crash and should be discarded rather than treated as a fatal error.

## Root cause
`readLog()` threw `IOException` when remaining bytes < length + 4 for the last entry. A crash during `appendToLog()` after writing the length header but before completing data+CRC leaves a partial entry.

## Proposed fix
Changed the truncation check from `throw` to `break` — truncated trailing entries are discarded. Valid entries before the truncation point are preserved.
