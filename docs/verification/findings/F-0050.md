# F-0050: Raft Inter-Node Transport Runs Plaintext Even When TLS Is "Enabled"

**Severity:** P1 (High)
**Phase:** V8 — Security Audit
**Status:** Closed — fix: TlsManager now threaded through TcpRaftTransport with fail-closed startup guard in ConfigdServer.start; test: io.configd.server.ConfigdServerTest::find0050_tcpRaftTransportExposesTlsManagerGetter and ::find0050_configdServerFailClosedCheckIsPresent

## Summary

`ConfigdServer.start()` only threads the `TlsManager` / `SSLContext` into the
HTTP API server. The real Raft peer transport (`TcpRaftTransport`) is
constructed with `null` for both the `tlsManager` and the
`sslContext`-producing parameter, so all Raft traffic — AppendEntries,
RequestVote, snapshot transfer, state-machine commands, auth tokens embedded
in commands, and any Ed25519-signed deltas — travels over plaintext TCP
even when the operator passes `--tls-cert` / `--tls-key` / `--tls-trust-store`
on the command line.

The "mTLS" claim in `docs/architecture.md`, `docs/prr/security-report.md`,
and ADR-0010 therefore only applies to the client-facing HTTP API, not to
cluster-internal traffic.

## Location

- `configd-server/src/main/java/io/configd/server/ConfigdServer.java:170-171`

```java
tcpTransport = new TcpRaftTransport(
        config.nodeId(), bindAddr, peerAddresses, null, null);
```

Both trailing `null` arguments are the `TlsManager` and the `MessageHandler`.
`TcpRaftTransport.createClientSocket` / `createServerSocket`
(`configd-transport/.../TcpRaftTransport.java:268-311`) fall through to the
plaintext `new Socket(...)` / `new ServerSocket()` branches whenever
`tlsManager == null`.

## Impact

- Any on-path attacker inside the pod network (sidecar, compromised node,
  mis-routed overlay) can read and forge Raft traffic: terms, votes, log
  entries, commit indices. That lets them partition the cluster, inject
  committed writes, or silently drop heartbeats.
- Ed25519 signatures on `ConfigDelta` do not compensate: Raft log entries
  themselves are unsigned and are trusted by `RaftNode` once the term/leader
  checks pass — both of which depend on peer identity being established by
  TLS client certs.

## Suggested Fix

- Pass the same `TlsManager` instance to the `TcpRaftTransport` constructor.
- Fail-closed: reject `config.tlsEnabled() == true` with a plaintext Raft
  transport at startup.
- Add an integration test that asserts `TlsManager` non-null on the Raft
  transport whenever `ServerConfig.tlsEnabled()` is true.

## Residual Risk Until Fixed

Operationally, any deployment that uses the bundled `ConfigdServer.start`
entry point cannot claim mTLS between cluster members regardless of how
certificates are provisioned.
