# F-0009: Linearizable read NOT linearizable — ReadIndex protocol started but never awaited

## Severity
P0

## Phase
V3

## Component
configd-server:ConfigdServer.java:316

## Invariant or target violated
INV-L1 (Linearizable writes/reads per Raft group) from consistency-contract.md

## Reproducer
1. Start a 3-node Raft cluster.
2. Issue a linearizable read to the leader immediately after a write on a different node.
3. Observe that the read handler calls `raftNode.readIndex()` but does not await quorum confirmation before serving the response.
4. The read may return stale data because the leader has not confirmed it still holds leadership via heartbeat quorum.

## Evidence
`LeadershipConfirmer` was implemented as `() -> raftNode.readIndex() >= 0` which calls `readIndex()` to start the ReadIndex protocol and checks if the returned ID is valid, but then discards the ID without ever calling `isReadReady()`. The read is served immediately to the client without waiting for heartbeat quorum confirmation or state machine catch-up. This means a stale leader (one that has been partitioned but does not yet know it) will serve reads that violate linearizability.

## Root cause
`LeadershipConfirmer` was `() -> raftNode.readIndex() >= 0` which calls `readIndex()` to start the protocol, checks if the returned ID is valid, then discards it without ever calling `isReadReady()`. The read is served immediately without waiting for heartbeat quorum confirmation or state machine catch-up.

## Proposed fix
Dispatch `readIndex()` and `isReadReady()` to the tick thread via `CompletableFuture`. Poll `isReadReady()` with 150ms timeout. Only serve read after confirmed.
