# F-0075: ADR-0009, ADR-0012, ADR-0014 claim Agrona/JCTools usage but no source file imports them

## Severity
P2 (Medium)

## Phase
V8 (dependency-claim re-audit)

## Component
- docs/decisions/adr-0009-java21-zgc-runtime.md (claims: Agrona off-heap, JCTools MPSC/SPSC)
- docs/decisions/adr-0012-purpose-built-storage-engine.md (claims: "Values > 4 KB stored off-heap via Agrona DirectBuffer", "Agrona MappedByteBuffer")
- docs/decisions/adr-0014-zgc-shenandoah-gc-strategy.md (claims: "Raft WAL segments: Written via Agrona MappedByteBuffer", "Off-heap slab allocator (Agrona DirectBuffer)", "JCTools MpscArrayQueue / SpscArrayQueue")

## Invariant or target violated
Doc-vs-code reconciliation. `pom.xml` declares `org.agrona:agrona:1.23.1` and `org.jctools:jctools-core:4.0.5` under `<dependencyManagement>`, but no source or test file imports either library:

```
$ grep -rn "import org.agrona\|import org.jctools" /home/ubuntu/Programming/Configd --include='*.java'
(no matches)
```

## Evidence
Performance.md §2 already partially concedes the gap: "JCTools SPSC/MPSC queues — lock-free, wait-free (**planned, not yet integrated**)." ADR-0009 and ADR-0014 make stronger, present-tense claims ("Raft WAL segments: Written via Agrona MappedByteBuffer", "MPSC queues: Client request ingestion ... JCTools MpscArrayQueue").

The actual WAL implementation (`configd-common/src/main/java/io/configd/common/FileStorage.java` — modified in Phase V3 per F-0011/F-0012) uses `java.nio.channels.FileChannel` with `force(true)` for fsync, not Agrona mapped buffers. Large values are stored on-heap, not via off-heap slabs.

## Root cause
ADR claims track the design intent, not the shipping implementation. No V8 pass caught this because the `pom.xml` check only verifies declared dependencies, not actual usage.

## Proposed fix
Add a V8-style "Not Implemented"/"Planned" note to the relevant subsections, analogous to the ADR-0016 reconciliation. Specifically:
- ADR-0009 Thread Model + "Off-heap storage via Agrona" → note that off-heap storage and JCTools queues are planned; current implementation uses on-heap byte arrays and `java.util.concurrent` queues.
- ADR-0012 §3 "Off-Heap Storage (Large Values)" → mark as not implemented.
- ADR-0014 §3 "Off-Heap Storage via Agrona" and §4 "Lock-Free Structures via JCTools" → mark as not implemented.

Alternatively, remove the unused Agrona/JCTools entries from `pom.xml`'s `<dependencyManagement>` until they are actually wired in.
