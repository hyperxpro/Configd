# F-0021: Linearizable read test bypasses F-0010 tick-thread dispatch, reintroducing data race in test code

## Severity
P2

## Phase
V3

## Component
configd-server/src/test/java/io/configd/server/ConfigdServerTest.java:372-391

## Invariant or target violated
F-0010 invariant: "ALL access to ReadIndexState must be from the tick thread
(no HTTP handler thread touching it)." The test that claims to verify the
F-0009 / F-0010 fix calls `raftNode.readIndex()`, `raftNode.isReadReady(...)`,
and `raftNode.completeRead(...)` *directly from the test thread*, not via
any executor. At the same time the RaftNode's tick thread may be driven by
`driver.tick()` elsewhere in the test harness. This recreates exactly the
LinkedHashMap data race the F-0010 fix eliminated, inside the test itself.

## Reproducer
Static read of test source at lines 369-391:
```
ConfigReadService readService = new ConfigReadService(reader, () -> {
    long readId = raftNode.readIndex();        // <-- test thread mutates pendingReads
    ...
    if (raftNode.isReadReady(readId)) {        // <-- test thread reads pendingReads
        raftNode.completeRead(readId);          // <-- test thread mutates pendingReads
        ...
```
No `readTickExecutor.execute(...)` wrapper — direct in-thread call.

## Evidence
Production `ConfigdServer.java:342-393` correctly dispatches every
`readIndex()` / `isReadReady(...)` / `completeRead(...)` invocation via
`readTickExecutor.execute(...)`. The test that exercises the "fixed"
behavior deliberately omits that dispatch, so the test cannot observe a
regression that reintroduces the bug, and the test itself contains the
race it is supposed to prove fixed.

## Root cause
When the F-0010 fix was introduced in the production path, the matching
test was not updated to reflect the new invariant. The confirmer lambda
in the test still mirrors the *old*, racy call pattern.

## Proposed fix
Rewrite the test confirmer to use the same `ScheduledExecutorService`
dispatch pattern as production, OR construct the `LeadershipConfirmer`
from the production factory helper rather than duplicating the logic.
Alternatively, add a Jcstress / Lincheck harness that pounds
`readIndex`/`isReadReady`/`completeRead` from multiple threads with the
tick thread running concurrently and asserts no CME / infinite loop.

## Status
Closed — fix: `ConfigdServerTest.java` confirmer now dispatches `readIndex()` / `isReadReady()` / `completeRead()` through a single-threaded tick executor (mirroring the F-0022/F-0023 production wiring) and uses the new `whenReadReady(readId, callback)` API instead of a test-thread poll loop; test: updated test still verifies the original F-0009 invariant (`linearizableRead` returns null when the node is not a confirmed leader) and passes under the new dispatch path.
