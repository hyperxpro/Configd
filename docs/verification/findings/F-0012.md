# F-0012: Missing directory fsync after WAL rewrite in RaftLog.truncateFrom()

## Severity
P1

## Phase
V3

## Component
configd-consensus-core:RaftLog.java:361

## Invariant or target violated
Log matching property (INV-3) across crashes — truncation during AppendEntries conflict resolution must be durable

## Reproducer
1. Start a 3-node cluster and induce a log conflict (e.g., partition the leader, let a new leader commit different entries, then heal the partition).
2. The follower with conflicting entries receives AppendEntries and calls `truncateFrom()` to remove the conflicting suffix.
3. `truncateFrom()` calls `rewriteWal()` which performs `renameLog()` to atomically replace the WAL file.
4. Immediately crash the machine (e.g., power loss) before the filesystem flushes directory metadata.
5. On restart, the old WAL file may still be present because the rename metadata was not persisted — the directory was never fsynced.
6. The node recovers with the pre-truncation log, violating the log matching property.

## Evidence
`truncateFrom()` calls `rewriteWal()` which does `renameLog()` but performs no directory fsync afterward. On Linux ext4, `rename()` updates directory metadata which may remain in the page cache indefinitely. The `compact()` method had a `sync()` call after its WAL rewrite, but `truncateFrom()` did not — an inconsistency indicating the fsync was simply forgotten.

## Root cause
`truncateFrom()` calls `rewriteWal()` which does `renameLog()` but no directory fsync. On Linux ext4, the rename metadata may not survive a crash. `compact()` had a `sync()` call but `truncateFrom()` did not.

## Proposed fix
Added `storage.sync()` after `rewriteWal()` in `truncateFrom()`.
