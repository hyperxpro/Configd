# F-0041: HybridClock.now()/receive() allocate a new HybridTimestamp (24 B/op) on every call

## Severity
P1

## Phase
V-perf

## Component
configd-common/src/main/java/io/configd/common/HybridClock.java:34,54,59

## Invariant or target violated
docs/performance.md — "Zero allocation on read path steady state".
HybridClock is not strictly "read path" (ADR-0007 notes HLC is a write-path clock) but it is called on every mutating RPC and on every HLC receive. The architecture claims HLC.now() < 40 ns p50 / < 100 ns p99 — when a GC occurs the allocation cost creates a tail spike much larger than 100 ns.

## Reproducer
```
java --enable-preview -cp <testkit+deps> org.openjdk.jmh.Main \
  io.configd.bench.HybridClockBenchmark -prof gc -f 1 -wi 3 -i 5 -bm avgt
```

## Evidence
```
HybridClockBenchmark.now                               fixed  avgt    5    21.353 ±   28.692   ns/op
HybridClockBenchmark.now:gc.alloc.rate.norm            fixed  avgt    5    24.000 ±    0.001    B/op
HybridClockBenchmark.now                              system  avgt    5   158.618 ±  118.017   ns/op
HybridClockBenchmark.now:gc.alloc.rate.norm           system  avgt    5    24.001 ±    0.001    B/op
HybridClockBenchmark.receive                           fixed  avgt    5    18.152 ±    9.940   ns/op
HybridClockBenchmark.receive:gc.alloc.rate.norm        fixed  avgt    5    24.000 ±    0.001    B/op
HybridClockBenchmark.receive                          system  avgt    5    78.867 ±  105.116   ns/op
```

Two separate issues:

**A. Steady-state 24 B/op allocation.** Every call to `now()`, `receive()`, or `current()` does `return new HybridTimestamp(lastWallTime, lastLogical);`. At 1.2 GB/s measured allocation rate this puts sustained pressure on the young-gen and is incompatible with a zero-alloc read/write path.

**B. `system` clock variant p50 ~158 ns (mean), target < 40 ns.** The OS clock call (`System.currentTimeMillis()`) inside a `synchronized` method dominates. `fixed` variant at 21 ns shows the HLC logic itself is fine — the budget is consumed by the real-clock call.

## Root cause
- `HybridTimestamp` is a heap class (not a value class / `@jdk.internal.ValueBased`). Escape analysis cannot eliminate the allocation because `Blackhole.consume()` forces the object to escape, but in production callers frequently persist the timestamp too (WAL records, response fields), so EA elimination is unlikely even outside benchmarks.
- `synchronized` prevents scalar replacement across threads; combined with the constructor this guarantees an allocation per call.

## Proposed fix
1. **Pack HLC into a `long`** — 48 bits wall-ms (enough until year 10895), 16 bits logical counter. Publish as a `VarHandle`-backed atomic long, eliminating both the `synchronized` block and the allocation. Return `long` from `now()`; keep `HybridTimestamp` as a decode helper.
2. If callers genuinely need an object, make `HybridTimestamp` a JDK 25 **value class** (`value record HybridTimestamp(...)`) so the JIT can flatten it on the stack.
3. Move `System.currentTimeMillis()` outside the synchronized block (read once, then CAS) to avoid serializing OS calls. Prefer a coarse-grained clock helper (VarHandle-published wall time updated by a background thread at ~1ms granularity) for the hot path.

## Status
Closed — fix: `HybridClock` rewritten to hold a single packed `long` (48-bit physical ms ‖ 16-bit logical counter) published via `VarHandle` CAS; `now()`, `receive(long)`, and `current()` now return a primitive `long` with zero allocation. Structured form (`nowStructured()`, `receive(HybridTimestamp)`, `currentStructured()`) retained for API compatibility and decodes the packed long on demand. `System.currentTimeMillis()` is read outside any lock. File: `configd-common/src/main/java/io/configd/common/HybridClock.java`. test: `configd-common/src/test/java/io/configd/common/HybridClockAllocationTest.java` (new) asserts ≤ 8 KB thread-allocated bytes over 10,000 `now()` and `receive(long)` calls via `com.sun.management.ThreadMXBean.getThreadAllocatedBytes` (baseline was ~240 KB); existing `HybridClockTest` updated to exercise the `long` API and the packed-encoding round-trip.
