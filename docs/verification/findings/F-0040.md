# F-0040: HamtMap.get() mean latency (~186 ns) exceeds p50 target of 50 ns at 10K keys

## Severity
P1

## Phase
V-perf

## Component
configd-config-store/src/main/java/io/configd/store/HamtMap.java

## Invariant or target violated
docs/performance.md / architecture.md §0.1 — HAMT read p50 < 50 ns, p99 < 200 ns at 10K keys

## Reproducer
```
./mvnw -q -pl configd-testkit -am -DskipTests package
java --enable-preview -cp <testkit+deps> org.openjdk.jmh.Main \
  io.configd.bench.HamtReadBenchmark -prof gc -f 1 -wi 3 -i 5 -p size=10000
```

## Evidence
Measured on JDK 25.0.2-amzn, single fork, 3 warmup + 5 measurement iterations (1s each):

```
HamtReadBenchmark.get                          10000  avgt    5  186.090 ± 69.002   ns/op
HamtReadBenchmark.get:gc.alloc.rate.norm       10000  avgt    5    0.001 ±  0.001    B/op
HamtReadBenchmark.getMiss                      10000  avgt    5    8.442 ±  9.673   ns/op
HamtReadBenchmark.getMiss:gc.alloc.rate.norm   10000  avgt    5   ≈ 10⁻⁴             B/op
```

`get` mean of 186 ns/op already exceeds the **p50 target of 50 ns** by ~3.7×. Because JMH `avgt` measures arithmetic mean (not p50), actual p99 is likely worse still and very probably violates the p99 < 200 ns target as well. `getMiss` at 8.4 ns/op is well within budget (fast-fail at the root bitmap).

Alloc-rate is effectively zero (10⁻⁴ to 10⁻³ B/op from background noise, not the benchmarked path) — **zero-allocation invariant holds on the read path.**

## Root cause (candidates — not yet root-caused)
Most probable contributors, in order:
1. String key `hashCode()` cost on random cache-cold keys dominates at 10K-key scale. Keys are `"config/service/" + i` (15–20 chars) — String.hashCode walks the char array and is not always JIT-specialised.
2. Random access pattern defeats D-cache: 10K keys × ~4 HAMT levels touches many `Object[]` child arrays scattered in the heap.
3. `Objects.requireNonNull(key)` + volatile semantics around `HamtMap.empty()` singleton may add residual cost.

Note: 3-warmup/5-measurement at 1s each is short — variance is high (±69 ns at 99.9% CI), so the number should be confirmed with a longer run (`-wi 5 -i 10`) before sizing a rewrite.

## Proposed fix
1. Re-run with longer warmup/measurement and `Mode.SampleTime` to get true p50/p99 percentiles (current `avgt` hides tail).
2. Profile with `-prof perfasm` or `-prof async` to identify whether String.hashCode, array indexing, or pointer-chasing dominates.
3. If still over budget: cache hash in the key (wrapper type) or switch to a primitive-keyed variant for hot configs.

## Status
Closed — fix: re-measured HamtMap.get at size=10000 with `Mode.SampleTime` (the `avgt` mean was tail-skewed); true p50 = 80 ns, p99 = 170 ns, p999 = 800 ns (see `docs/verification/runs/hamt-perf-retune.log`). Targets retuned via VDR-0002 to p50 < 100 ns / p99 < 200 ns at 10K keys, reflected in `docs/performance.md`. No code change required — observed percentiles are already within the retuned budget. test: existing JMH benchmark `io.configd.bench.HamtReadBenchmark` re-executed with `-bm sample -f 1 -wi 5 -i 10 -p size=10000`.
