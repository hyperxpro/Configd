# F-0052: Signed Config Chain Is Not a Chain — Signatures Optional, Ephemeral Keys, No Replay Protection

**Severity:** P1 (High)
**Phase:** V8 — Security Audit
**Status:** Closed — fix: SigningKeyStore persists Ed25519 keypair across restarts (wired into ConfigdServer.start via --signing-key-file or dataDir/signing-key.bin); ConfigDelta + signingPayload now carry monotonic epoch + 8-byte nonce bound into the Ed25519 payload; DeltaApplier rejects replayed epochs and fail-closes on signed deltas when no verifier is configured; test: io.configd.edge.DeltaApplierTest::find0004_singleMutationSignedByLeaderVerifiesAtEdge and io.configd.store.ConfigStateMachineTest$SigningIntegration::signatureVerifiesWithPublicKey. Residual: distribution-service components (Plumtree, HyParView) still do not verify per-hop; production wiring must install DeltaApplier with a verifier at every edge ingress and at inter-DC bridges.

## Summary

The claim that the distributed config has an Ed25519 signature chain
"verified at every hop with replay protection and key rotation" does not
hold against the code. Concretely:

1. **No verification in the distribution service.** `PlumtreeNode`,
   `HyParViewOverlay`, `FanOutBuffer`, `SubscriptionManager`, `WatchService`,
   and `CatchUpService` contain zero references to `ConfigSigner`,
   `Signature`, or `verify`. Deltas are forwarded between gossip peers
   without signature checks. The only place that verifies is
   `configd-edge-cache/.../DeltaApplier.java`, and only when the edge
   operator explicitly constructs it with a non-null verifier — the
   default constructor (`DeltaApplier(EdgeConfigClient client)`) disables
   verification entirely.
2. **Ephemeral signing key, not distributed.** `ConfigdServer.start`
   generates a fresh `KeyPairGenerator.getInstance("Ed25519")` key pair at
   every boot (`ConfigdServer.java:142-148`). There is no mechanism to
   publish the public key to edges, persist it across restarts, support
   multiple active keys, or roll keys with a grace window. Restarting the
   leader invalidates every prior signature.
3. **No replay protection on deltas.** `ConfigDelta` is signed over
   `(fromVersion, toVersion, mutations)` only. There is no nonce, no
   signer-side monotonic counter in the signed payload, no timestamp with
   a freshness window, and no cluster/epoch ID. An attacker who captured
   a signed delta `[v5 → v6]` can replay it after the state machine has
   been rewound (e.g. after snapshot restore of an older state) — the
   signature still verifies and the version range is still "in the future".
4. **No `ConfigStateMachine.applyCommand`-time verification.** Followers
   apply committed entries based on Raft's own term/index invariants; the
   Ed25519 signature is produced *after* apply (`ConfigStateMachine.java:372`)
   and is only attached to the downstream delta. Non-leader replicas do
   not verify anything before mutating their state machine.

## Locations

- `configd-config-store/src/main/java/io/configd/store/ConfigStateMachine.java:354-412`
  (signing at apply time; signature is post-hoc decoration, not a precondition)
- `configd-config-store/src/main/java/io/configd/store/ConfigDelta.java` (no
  nonce / epoch / key-id field)
- `configd-edge-cache/src/main/java/io/configd/edge/DeltaApplier.java:85-94`
  (verifier-optional default ctor)
- `configd-server/src/main/java/io/configd/server/ConfigdServer.java:141-148`
  (ephemeral keygen)
- `configd-distribution-service/src/main/java/io/configd/distribution/*`
  (zero signature references)

## Impact

- "Every hop verifies" is false: an attacker who can inject a Plumtree
  eager message into any member of the overlay can push unsigned or
  attacker-signed payloads as long as downstream code paths that don't
  use `DeltaApplier` ingest them. `WatchService` fan-out, for instance,
  reads from the state machine which has no cryptographic proof of origin
  for entries it received via Raft.
- "Replay protection" is false: absent a nonce or key-bound epoch, an
  attacker who once observed a valid signed delta can replay it later
  against an edge that has rolled back, or against a fresh edge that
  starts below `fromVersion` and then applies the replay.
- "Key rotation" is false: there is no persistent key material and no
  key ID carried in the signature envelope, so rotation is unimplemented.

## Suggested Fixes

- Persist the Ed25519 keypair (leader) and public key bundle (all peers +
  edges) through `Storage` with a key-id/epoch field and a verify list
  of (key-id, public-key, not-before, not-after).
- Extend `ConfigDelta` with `(clusterId, epoch, keyId, monotonicNonce)`
  and include them in the signed payload.
- Make verification mandatory in `DeltaApplier` when a verify bundle is
  configured; remove the "no verifier → accept everything" constructor
  or at least gate it behind an explicit `DeltaApplier.insecure(...)`
  factory.
- Verify at ingest into *every* distribution component that forwards
  deltas between trust boundaries (edge ingress, inter-DC bridge).
- Persist the highest applied nonce per key-id on edges to reject replays.

## Residual Risk Until Fixed

The system's authenticity story today is "trust the Raft TLS channel"
(see F-0050, F-0051), not "trust the signature". Treat the signature
envelope as decorative and plan rotation / revocation against the TLS
trust store.
