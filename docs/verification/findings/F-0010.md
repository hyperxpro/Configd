# F-0010: Data race on ReadIndexState from HTTP handler threads

## Severity
P0

## Phase
V3

## Component
configd-server:ConfigdServer.java:316 and configd-consensus-core:ReadIndexState.java:39

## Invariant or target violated
Java Memory Model safety — concurrent unsynchronized writes to LinkedHashMap

## Reproducer
1. Start a Raft leader node and issue concurrent linearizable read requests from multiple HTTP clients.
2. Each HTTP handler thread calls `raftNode.readIndex()`, which writes to `ReadIndexState.pendingReads` (a plain `LinkedHashMap`).
3. Simultaneously, the tick thread calls `confirmPendingReads()` which reads and writes the same `pendingReads` map.
4. Under load, observe `ConcurrentModificationException` or silent data corruption in the `LinkedHashMap` due to unsynchronized concurrent access.

## Evidence
`ReadIndexState.pendingReads` is a plain `LinkedHashMap` with no synchronization. HTTP handler threads invoke `raftNode.readIndex()` which inserts entries into `pendingReads`. The single tick thread also reads and modifies `pendingReads` via `confirmPendingReads()`. The Java Memory Model provides no guarantees for concurrent unsynchronized access to `LinkedHashMap` — this can cause infinite loops in the hash bucket chains, lost entries, or `ConcurrentModificationException`.

## Root cause
`raftNode.readIndex()` was called from HTTP handler threads which writes to `ReadIndexState.pendingReads` (plain `LinkedHashMap`). The tick thread also reads/writes `pendingReads` via `confirmPendingReads()`. No synchronization.

## Proposed fix
All `ReadIndexState` access now dispatched to the single tick thread via the `ScheduledExecutorService`.
