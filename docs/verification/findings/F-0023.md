# F-0023: Read-dispatch executor is shared with consensus tick loop and TLS reload — head-of-line blocking

## Severity
P2

## Phase
V3

## Component
configd-server/src/main/java/io/configd/server/ConfigdServer.java:313-317, 411-454

## Invariant or target violated
Liveness of linearizable reads and consensus progress. The F-0009 /
F-0010 fix dispatches `readIndex()`, `isReadReady()`, and `completeRead()`
to the `readTickExecutor`. The same single-threaded executor also runs:
(a) the consensus tick every 10ms (`driver.tick()`, heartbeat/election
timers, propagation monitor, watch service, plumtree tick, compaction),
and (b) TLS certificate reload every 60s.

A long tick (e.g., compaction every 1000 ticks ≈ 10s, or a GC pause
during tick) or a slow TLS reload I/O causes:
- Queued `isReadReady` tasks miss their HTTP 150ms deadline, forcing
  linearizable reads to return `false` and the API to respond 503.
- Queued `readIndex()` tasks also miss deadline → same 503.
- While reads are queued, heartbeats are not sent → a transient stall
  can cause followers to time out and start a new election (cluster
  availability dip).

## Reproducer
Static reading of `ConfigdServer.java`:
- Line 313: `Executors.newSingleThreadScheduledExecutor(...)` (single
  worker).
- Line 347, 366, 378, 385: all linearizable-read dispatches submit to
  this same executor.
- Line 413: `tickExecutor = readTickExecutor;` — literal alias.
- Line 421: `tickExecutor.scheduleAtFixedRate(...)` — consensus tick.
- Line 446: `tickExecutor.scheduleAtFixedRate(...)` — TLS reload.

A bursty read workload can force `isReadReady` dispatches to queue
behind the scheduled tick task; because the executor is FIFO and the
scheduled task is submitted with a fixed-rate trigger, starvation in
either direction is possible.

## Evidence
```java
ScheduledExecutorService readTickExecutor = Executors.newSingleThreadScheduledExecutor(...);
...
ScheduledExecutorService tickExecutor = readTickExecutor;   // alias, same thread
tickExecutor.scheduleAtFixedRate(() -> { driver.tick(); ... }, 10, 10, MS);
tickExecutor.scheduleAtFixedRate(() -> tlsManager.reload(), 60_000, 60_000, MS);
```

## Root cause
The fix re-used the tick executor to guarantee serialization with
ReadIndexState mutations. This is correct for memory-model safety but
conflates two distinct latency domains (consensus progress vs. read
confirmation) onto one CPU-bound thread plus one potentially
I/O-blocking reload.

## Proposed fix
Option A (preferred): keep ReadIndexState on the tick thread but publish
a lock-free completion path — the tick thread flips a `CompletableFuture`
per read when `isReady` becomes true, and the HTTP thread awaits that
future without re-dispatching (see F-0022). This eliminates the queue
entirely.

Option B: Move TLS reload to a dedicated single-thread executor (reload
is I/O-bound and unrelated to consensus invariants). Keep reads + tick
on one thread.

Either way, add a monitoring counter for `tick_queue_depth` / oldest
queued task age; alert when > 10ms (i.e., tick period).

## Status
Closed — fix: `ConfigdServer` split the shared scheduled executor into three single-threaded executors: `tickExecutor` (consensus tick + ReadIndexState — preserves F-0010 invariant), `readDispatchExecutor` (marshals HTTP read dispatches to the tick thread), and `tlsReloadExecutor` (isolates 60s TLS cert reload I/O); `shutdown()` tears them down in order read-dispatch → tick → tls-reload; test: `ConfigdServerTest.tlsReloadDoesNotBlockTickLoop` runs a 500ms blocking task on the TLS executor and asserts the tick executor still fires ≥20 ticks in 400ms.
