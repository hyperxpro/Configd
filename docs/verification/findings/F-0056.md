# F-0056: maven-jar-plugin Version Unpinned in configd-server/pom.xml

**Severity:** P3 (Low)
**Phase:** V8 — Security Audit
**Status:** Closed — fix: maven-jar-plugin pinned to 3.4.2 in the root pom.xml pluginManagement; test: io.configd.server.ConfigdServerTest::find0056_mavenJarPluginIsPinned

## Summary

`./mvnw dependency:tree` warns:

```
[WARNING] 'build.plugins.plugin.version' for
org.apache.maven.plugins:maven-jar-plugin is missing.
@ configd-server/pom.xml line 40, column 21
```

Every third-party library on the classpath *is* pinned to an explicit
release (no `SNAPSHOT`, `LATEST`, or `RELEASE` entries were found in
the effective tree apart from the project's own `0.1.0-SNAPSHOT` artifact
id, which is acceptable for an in-development repo). So supply-chain
hygiene is broadly good — but an unpinned build plugin opens a minor
reproducibility hole: the jar packer can change between runs as Maven
selects whatever `maven-jar-plugin` version it resolves.

## Suggested Fix

Pin the plugin explicitly in `configd-server/pom.xml`:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-jar-plugin</artifactId>
  <version>3.4.1</version>
</plugin>
```

Consider adding `maven-enforcer-plugin` with `requirePluginVersions` so
future drift is a hard build error rather than a warning.

## Note

Project artifacts are published as `0.1.0-SNAPSHOT` (parent + 12 modules).
That is a project-version concern, not a third-party supply-chain concern,
so it is not logged as a supply-chain finding here. When cutting 1.0 it
should become a pinned release version.
