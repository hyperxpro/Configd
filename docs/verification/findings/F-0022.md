# F-0022: Per-poll CompletableFuture + lambda allocation on linearizable read path

## Severity
P3

## Phase
V3

## Component
configd-server/src/main/java/io/configd/server/ConfigdServer.java:342-393

## Invariant or target violated
"No allocation in steady state on the hot read path." Linearizable reads
are opt-in but are the documented Quicksilver consistency primitive; any
production deployment that uses `?consistency=linearizable` is on the hot
read path for that class of request.

## Reproducer
Static read of `ConfigdServer.java:342-393`:
- Per request: 1 `CompletableFuture<Long>` + 1 `Runnable` capture (`readIdFuture`,
  `raftNode` reference) — 2 allocations before the first poll.
- Per poll iteration inside the `while (System.nanoTime() < deadline)` loop:
  1 additional `CompletableFuture<Boolean>` + 1 `Runnable` capture
  (`readyFuture`, `readId`, `raftNode`). With a 150ms window and 1ms sleep
  between polls, this is up to ~150 allocations per read.
- On ready/timeout: 1 additional `Runnable` for `completeRead`.
- The repeated `get(remainingMs, TimeUnit.MILLISECONDS)` call autoboxes the
  `Boolean` return value on every completion.

Each allocation is short-lived, but under sustained linearizable-read load
these touch the nursery on every hop and perturb tail latency (the SLO
target).

## Evidence
Excerpt:
```java
while (System.nanoTime() < deadline) {
    var readyFuture = new java.util.concurrent.CompletableFuture<Boolean>();
    readTickExecutor.execute(() -> {
        try {
            readyFuture.complete(raftNode.isReadReady(readId));
        } catch (Throwable t) {
            readyFuture.completeExceptionally(t);
        }
    });
    long remainingMs = TimeUnit.NANOSECONDS.toMillis(deadline - System.nanoTime());
    if (remainingMs <= 0) break;
    boolean ready = readyFuture.get(remainingMs, TimeUnit.MILLISECONDS);
    ...
    Thread.sleep(1);
}
```

## Root cause
The F-0009 fix was implemented with a "poll by dispatch" pattern that
allocates a fresh future and lambda on every iteration. A single-shot
design (dispatch once, have the tick thread complete the future when
`isReady` flips true, bounded by a `ScheduledFuture` timeout) would
allocate O(1) per read instead of O(polls) per read.

## Proposed fix
Replace the polling loop with a single completion-driven future:
register the read in a tick-thread-local map from `readId` to a
pre-allocated `CompletableFuture<Boolean>`, have `confirmPendingReads()`
complete the future on the tick thread when `isReady` flips true, and
arm a single `readTickExecutor.schedule(..., 150, MILLISECONDS)` to
complete-exceptionally on timeout. One future, one scheduled task per
read. This also removes `Thread.sleep(1)` from the HTTP path.

## Status
Closed — fix: added `RaftNode.whenReadReady(readId, Runnable)` one-shot callback API fired from the tick thread in `confirmPendingReads()`, `applyCommitted()`, and `becomeFollower()`; `ConfigdServer` now allocates exactly one `CompletableFuture` per linearizable read and completes it via the callback — replacing the per-poll allocation loop; test: `ConfigdServerTest.linearizableReadAllocatesAtMostOneFuturePerRead` asserts exactly 1 future per call across 10 reads (was ~150 under stall).
