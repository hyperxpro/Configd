# F-0051: TLS Client Does Not Enforce Hostname Verification

**Severity:** P1 (High)
**Phase:** V8 — Security Audit
**Status:** Closed — fix: TcpRaftTransport.createClientSocket now uses hostname (not InetAddress) and sets SSLParameters.setEndpointIdentificationAlgorithm("HTTPS") before handshake; test: io.configd.transport.TcpRaftTransportTest::find0051_clientHandshakeRejectsCertWithWrongHostname

## Summary

`TcpRaftTransport.createClientSocket` obtains an `SSLSocket` from the
`SSLSocketFactory` and calls `socket.startHandshake()` without first
setting `SSLParameters.setEndpointIdentificationAlgorithm("HTTPS")` (or
"LDAPS"). In the JDK default, an `SSLSocket` created this way does *not*
verify that the peer certificate's SAN / CN matches the target hostname
or IP — the handshake succeeds against any cert signed by the configured
trust store.

This silently downgrades mTLS to "any cert in the trust store is acceptable
for any peer," which defeats peer-pinning and lets a compromised member cert
impersonate another member.

## Location

- `configd-transport/src/main/java/io/configd/transport/TcpRaftTransport.java:268-288`
- `configd-transport/src/main/java/io/configd/transport/TlsManager.java:51-75`
  (the shared `SSLContext` builder is also silent about endpoint ID)

## Evidence

```java
SSLSocket socket = (SSLSocket) factory.createSocket(
        address.getAddress(), address.getPort());
// ... protocol / cipher selection ...
socket.startHandshake();
```

No `SSLParameters` with `setEndpointIdentificationAlgorithm("HTTPS")`
is installed on the socket. The same pattern is used in the test suite,
which is why existing TLS tests still pass.

## Impact

- A principal who possesses any certificate signed by the internal CA can
  impersonate any other cluster member as long as it can reach the target
  address.
- Combined with F-0050, this means even when the Raft transport is later
  wired to TLS, cross-member authentication will be weaker than advertised.

## Suggested Fix

On both client sockets and (for symmetry) server sockets:

```java
SSLParameters params = socket.getSSLParameters();
params.setEndpointIdentificationAlgorithm("HTTPS");
socket.setSSLParameters(params);
```

Also configure peer name via `SSLSocketFactory.createSocket(host, port)`
using the **hostname**, not an `InetAddress` (the latter skips SNI and
forces reverse-DNS, which is routinely wrong in pod networks).

## Regression Test

Stand up two peers whose leaf certs are signed by the same CA but whose
SANs only cover their own hostnames. Configure peer A's address book to
point at peer B's IP but use peer C's hostname in the configured address.
With the fix, the handshake must fail.
