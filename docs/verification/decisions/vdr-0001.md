# VDR-0001: ReadResult allocation at API boundary is accepted; zero-alloc scope limited to leaf HAMT

## Context

Finding F-0042 flagged that `VersionedConfigStore.get()` and
`LocalConfigStore.get()` allocate a fresh `ReadResult` (~24 B) on every hit.
The docs/performance.md target "Zero allocation on read path steady state" is
strictly violated by this per-hit allocation, even though the miss path is
genuinely zero-alloc (pre-allocated `NOT_FOUND` singleton).

## Options considered

1. **Document-and-accept.** Rewrite the performance claim to read "zero
   allocation in the leaf HAMT lookup; one ~24 B `ReadResult` wrapper per hit
   at the API boundary (unavoidable value-object return on Java reference
   types)". Cheap, clear, honest.
2. **Primitive-out-param API.** Add `int getInto(String key, byte[] dstBuf,
   long[] versionOut)` that writes into caller-supplied buffers and returns
   the value length (-1 on miss). Keeps the ergonomic `ReadResult` form for
   external callers but offers a zero-alloc path for throughput-critical
   internal call sites.
3. **Value class.** Convert `ReadResult` to a JDK 25 value class so the JIT
   can scalar-replace on stack. Viable long-term but blocked by preview-status
   of value classes and brittleness across call-site shapes (the record may
   escape into a gRPC response, a `Map`, etc., at which point scalar
   replacement fails anyway).

## Decision

Adopt **options 1 + 2** together.

- Update `docs/performance.md` / `docs/consistency-contract.md` wording so the
  invariant accurately describes where allocation occurs.
- Add `VersionedConfigStore.getInto(...)` and `LocalConfigStore.getInto(...)`
  primitive-friendly APIs for hot callers that want strictly zero allocation
  on both hit and miss paths.
- Keep `ReadResult.found(...)` and the `get(String)` API unchanged; mark the
  per-hit allocation as an accepted design cost.
- Do **not** convert `ReadResult` to a value class until the JDK 25 value
  class / Valhalla story stabilises.

## Consequences

- Performance invariant claim is now accurate: zero allocation in the leaf
  structure (HamtMap.get), at most one 24-byte wrapper at the API boundary.
- Throughput-critical internal hot loops (e.g. delta propagation, bulk
  replay) can switch to `getInto(...)` and achieve strict zero-alloc.
- External consumers of the `ReadResult` API pay no cost.

## Status

Accepted — 2026-04-16.
